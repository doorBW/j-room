# insta-j-room-create-review — 후기 카드뉴스 제작

> j-room 인스타그램의 **후기(review)** 카테고리. 직접 체험한 **1개 테마**를 깊이 다룬다.
> 레퍼런스: `pipeline/data/review-sample.json` (레이아웃 데모)

---

## 0. 핵심 원칙

- **1테마 깊이**: 추천(여러 테마 얕게)과 반대. 한 테마를 자세히.
- **스포일러 절대 금지**: 자물쇠/힌트/정답/반전 노출 X. 스토리 1/3 시점까지만.
- **별점 기준 일관성**: 한 번 정한 기준을 계속 사용 (인플레이션 주의). → [docs/BRAND_GUIDE.md](../../../docs/BRAND_GUIDE.md)
- **솔직함**: 아쉬운 점도 완곡하지만 정직하게.
- 발행 슬롯: 격주 2주차 토요일(+1주차 보너스). 비정기 — 다녀온 만큼.

---

## 1. 카드 구조 (7장)

| # | type | 역할 |
|---|------|------|
| 1 | `cover` | 표지 — 매장 〈테마〉 + 후킹 |
| 2 | `ratings` | 별점 5종 (난이도/공포도/추리비중/추천도 + 권장인원) |
| 3 | `intro` | 👍 좋았던 점 |
| 4 | `intro` | 🤔 아쉬운 점 |
| 5 | `intro` | 이런 분께 (추천/비추천) |
| 6 | `store` | 매장 정보 |
| 7 | `cta` | 비슷한 테마 추천 유도 등 |

> 카드2는 추천과 달리 **별점(ratings)** — 쫄보 자기소개 아님. 쫄보 브랜딩은 표지 kicker("늙크크가 다녀온").

---

## 2. data JSON 스키마

cover/intro/store/cta는 추천과 동일. 신규는 `ratings`.

### cover
```json
{ "type": "cover", "kicker": "늙크크가 다녀온", "title": "매장명", "titleAccent": "〈테마명〉", "hook": "후킹 한 줄 👀", "sub": "직접 플레이한 솔직 후기 (스포 없음)" }
```

### ratings (카드 2)
```json
{
  "type": "ratings",
  "poster": "assets/images/<매장>/<테마>.jpg",  // 선택
  "store": "매장명",
  "name": "테마명",
  "quote": "스포 없는 한 줄 총평",
  "ratings": [
    { "label": "난이도", "score": 4 },
    { "label": "공포도", "score": 2 },
    { "label": "추리비중", "score": 5 },
    { "label": "추천도", "score": 5 }
  ],
  "players": "2~3인"
}
```
- score 1~5 정수 → 별바 자동(채운=빨강, 빈=회색). 우측에 숫자.

### intro (좋았던 점 / 아쉬운 점 / 이런 분께)
```json
{ "type": "intro", "title": "👍 좋았던 점", "points": ["...", "..."] }
{ "type": "intro", "title": "이런 분께", "subtitle": "추천 / 비추천", "points": ["✅ ...", "❌ ..."] }
```

### store / cta
추천과 동일.

## 3. 별점 기준 (일관 유지 — BRAND_GUIDE)

| 항목 | 1점 | 5점 |
|------|-----|-----|
| 난이도 | 입문자 OK | 베테랑도 힘듦 |
| 공포도 | 안 무서움 | 비명 보장 |
| 추리비중 | 활동/액션 위주 | 머리 계속 |
| 추천도 | 굳이 | 무조건 |

> "공식 표기 vs 체감"이 다르면 후기에선 **체감 기준**으로. (공식과 다르면 본문에 언급 가능)

---

## 4. 워크플로우

1. **정보**: 사용자(다녀온 4인방)가 별점·좋았던점·아쉬운점·추천대상 제공. 공식 정보(가격/위치/시간)는 웹 확인.
2. **스포 체크**: 좋았던점/아쉬운점에 스포 없는지 확인.
3. **포스터**(선택): 매장 공식 → `assets/images/<매장>/`, 영문명, SOURCES.md 기록, gitignore.
4. **data JSON** 작성 → `node render.js data/review-<매장>-<테마>.json`.
5. **검토**: 별점/오타/스포/footer. 캡션 작성.
6. content/drafts → published, git commit.
